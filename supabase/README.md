## iOS app: PostgREST vs current Express API

### Response shape

The Express API returns a wrapper:

```json
{ "data": [...], "total": 0, "page": 1, "limit": 20, "totalPages": 0 }
```

PostgREST returns a **bare JSON array** of rows, e.g. `GET /rest/v1/restaurants?city=eq.Jeddah`. Optional totals use the `Content-Range` header with `Prefer: count=exact`.

Update Swift stores (e.g. `RestaurantStore`) to decode `[Restaurant].self` (or a thin adapter), not `PaginatedResponse<Restaurant>`.

### Base URL and headers

- Base: `https://fltfmcqfoftzjhxxpsss.supabase.co/rest/v1`
- Headers on every request: `apikey: <anon key>`, `Authorization: Bearer <anon key>` (for public reads).
- Query filters: PostgREST syntax (`city=eq.Jeddah`, etc.), not `?city=Jeddah` unless you use `eq`/`ilike` as appropriate.

### iOS app (physical device)

The app reads **`MRASEM_SUPABASE_REST_URL`** and **`MRASEM_SUPABASE_ANON_KEY`** from `Info.plist`. When both are set, **restaurants**, **activities**, and **season events** load from Supabase (no Mac required). Copy the **anon** key from Supabase → **Project Settings → API**. Apply SQL migrations so tables exist and RLS allows `anon` `SELECT` on those tables.

**Vercel** deploys the **Next.js admin** only; it does not replace this REST URL for the iPhone app.

### Bookings: read path gap

RLS in this repo allows:

- **anon**: `INSERT` on `bookings` (mobile creates bookings)
- **authenticated**: `SELECT` and `UPDATE` on `bookings` (admin)

There is **no `SELECT` for anon** on `bookings`. The iOS app today loads “My reservations” with a **JWT** via `GET /api/mobile/bookings` on Express. If you only use the **anon** key on the device, users **cannot** list their own bookings through PostgREST until you add one of:

- **Supabase Auth** (e.g. phone) and RLS tying rows to `auth.uid()`, or  
- A **backend/Edge Function** that validates the device session and queries with the **service role**, or  
- Keep Express (or a BFF) for booking reads until auth is migrated.

Plan booking inserts to include `user_phone` (and other required columns) to match the `bookings` table.

### Table names

- Season events table: **`season_events`** (snake_case). iOS `SeasonEvent` + `JSONDecoder.keyDecodingStrategy = .convertFromSnakeCase` aligns with default PostgREST keys.

### `public.users`

This is the **app** phone book, not `auth.users`. Admin auth uses Supabase Auth separately.
